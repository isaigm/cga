A Full-Stack Embedded Systems project featuring a custom EDA tool written in **Rust** that dynamically generates synthesizable, massively parallel **VHDL** code to solve the NP-Hard N-Queens problem using a **Cellular Genetic Algorithm (CGA)**. 

The project includes a Rust workspace containing the hardware generator, a software benchmarking tool, and a frontend that listens via UART to display the hardware-accelerated solution on a visual chess board in real-time.

## 🧠 Architectural Highlights
* **Massively Parallel Spatial Architecture:** A central FSM orchestrates a 2D grid of 30 independent, custom-built processing cores (cells). Instead of relying on instruction sets (like SIMD), the algorithm is hardwired into the silicon, evaluating the entire population simultaneously.
* **Toroidal Mesh Topology:** Cells are connected to their North, South, East, and West neighbors wrapping around the edges (Pac-Man style), ensuring continuous genetic diversity without boundary edge-cases.
* **Hardware Unrolling & Multi-cycle Datapath:** The critical path is managed by a highly optimized multi-cycle FSM. The $O(N^2)$ fitness evaluation (checking all row and diagonal attacks) is fully unrolled into pure combinational logic, allowing the hardware to calculate all queen collisions in a **single clock cycle** while successfully achieving Timing Closure at 100 MHz.
* **Optimized Distributed PRNG:** Each cell contains a local LFSR initialized with a unique, deterministically generated seed from Rust. To maximize LUT efficiency, SIPO buffers were eliminated, and random entropy is extracted directly from the LFSR's internal state.

## ⚡ Hardware vs. Software Performance Showdown

To demonstrate the power of Spatial Parallelism, the algorithm was benchmarked running 30 consecutive executions against a modern CPU. While the CPU runs at a massive 5.0 GHz, it is severely bottlenecked by instruction fetching, memory hierarchy, and sequential evaluation loops. 

The FPGA running at 100 MHz evaluates all 30 cells instantly in physical silicon, completely obliterating the CPU's execution time.

| Metric | Software Benchmark (AMD Ryzen 5 8645HS) | Hardware Accelerator (Xilinx Artix-7 XC7A35T) | Hardware Advantage |
| :--- | :--- | :--- | :--- |
| **Clock Speed** | Up to 5.0 GHz | **100 MHz** | - |
| **Concurrency** | Sequential / Threaded | **30 Custom Parallel Cores** | - |
| **Fastest Convergence** | 70.7 µs | **1.44 µs** (144 cycles) | **~49x Faster** |
| **Worst-Case Stagnation** | 17.15 ms | **0.708 ms** (70,836 cycles) | **~24x Faster** |

## 📂 Workspace Structure

This project is managed as a Cargo Workspace containing three main crates:

```text
.
├── Cargo.toml                 # Workspace definition
├── frontend/                  # UART Listener & Visualizer
│   └── src/main.rs            # Reads serial data from the FPGA and draws the N-Queens board
│
├── cpu_cga/                   # CPU Benchmark Tool
│   └── src/main.rs            # Equivalent software implementation of the CGA (runs 30 times for profiling)
│
└── generator/                 # The Hardware Compiler (EDA Tool)
    ├── src/main.rs            # Rust script that calculates topology and writes VHDL
    └── vhdl/                  # Output directory for Vivado
        │
        ├── [ Dynamically Generated by Rust ]
        │   ├── cga.vhd            # Top-level FSM, Toroidal Mesh wiring, and UART control
        │   ├── cell.vhd           # The Processing Element (Crossover, Mutation, Fitness)
        │   ├── cell_comparator.vhd# Combinational Tournament Tree
        │   └── n_queens_types.vhd # Dynamic bit-width definitions
        │
        └── [ Fixed Hardware IPs ]
            ├── lfsr.vhd           # Linear-Feedback Shift Register (PRNG)
            └── uart.vhd           # UART Transmitter for real-time telemetry
```

## ⚠️ Hardware Limitations & Scaling ($N=10$)

The current Rust generator is configured to output a 5x6 grid (30 cores) solving the 10-Queens problem. Synthesizing this specific configuration consumes 18,702 LUTs and 12,929 FFs (nearly 90% of the logic capacity) of a Xilinx Artix-7 XC7A35T (Basys 3 board). The fitness function for N-Queens requires $O(N^2)$ parallel comparators per cell to evaluate row and diagonal attacks in a single clock cycle.

> **Note:** Generating the VHDL for larger grids or $N > 10$ Queens works perfectly in software, but synthesizing it will result in routing congestion and resource exhaustion on the Basys 3. Deploying larger problem sizes requires migrating to a larger FPGA (e.g., Kintex-7 or Virtex Ultrascale+).

##  Getting Started

### 1. Run the CPU Benchmark
To see the software-based execution times (30 consecutive runs):
```bash
cargo run -p cpu_cga --release
```

### 2. Generate the Hardware (VHDL)
To generate the structural VHDL files for the FPGA:

```bash
cargo run -p generator
```
Once generated, import the generator/vhdl/ folder into a Xilinx Vivado project, assign your board's constraints (Clock, Reset Button, and UART TX pin), and generate the bitstream.

### 3. Run the Visualizer (Frontend)
Once the FPGA is programmed and connected via USB, start the Rust frontend to listen to the serial port:
```bash
cargo run -p frontend

```
Press the "Start" button on your FPGA. The hardware will compute the solution at 100 MHz, transmit the winning chromosome via UART, and the Rust frontend will render the 10x10 chessboard with the safe queen positions.
